# Sprint 014: DOT File Upload on Create Page

## Overview

The Create view gives users two ways to produce a pipeline: generate one from a natural language prompt, or type/paste DOT source directly into the textarea. Users who already have a `.dot` file on disk — hand-authored, generated by the CLI, or downloaded from GitHub — currently must open the file in an editor, copy all its contents, switch to the browser, and paste them. This friction makes the "load and run" workflow unnecessarily cumbersome and discourages file-based pipelines.

This sprint adds an **Upload .dot** button to the run-row of the Create view (alongside Cancel and Run Pipeline). Clicking it opens the browser's native file picker filtered to `.dot` files. Once a file is selected, its text content immediately loads into the "Generated DOT" textarea, the NL prompt is cleared to signal a clean file-based state, and the graph preview auto-renders. The Run Pipeline button becomes enabled. The original filename is preserved and used as `fileName` when submitting the pipeline, so artifacts and logs are labelled with the actual file name rather than the generic `"generated.dot"`.

The implementation is pure client-side: one hidden `<input type="file">`, one button, and a `FileReader` handler in `WebMonitorServer.kt`. The existing `FileReader` pattern from the Import ZIP modal is the direct model. No new server routes, no new server handlers, no new Gradle dependencies. All existing Create view workflows — NL generate, manual paste, and iterate — continue to work exactly as before.

## Use Cases

1. **Load a pre-authored pipeline**: A user has written `deploy-container.dot` locally. They open the Create view, click "Upload .dot", select the file, and see the graph render immediately. They click Run Pipeline — the pipeline starts with `fileName = "deploy-container.dot"`.

2. **CLI/export roundtrip**: A developer ran `attractor dot generate --output pipeline.dot` from the terminal. They open the web UI, upload `pipeline.dot` to visually preview and run it without copying anything.

3. **Quick iteration from a downloaded example**: A user downloads `examples/branching.dot` from GitHub. They upload it to the Create view, see the graph, edit a few lines in the DOT textarea, and run the modified version.

4. **Upload then tweak**: A user uploads a `.dot` file, then edits the DOT textarea before running. After uploading they can also type a new NL description and click Iterate to further refine via LLM.

5. **Upload an invalid file**: A user accidentally selects a `.dot` file with a syntax error. The DOT loads into the textarea and the graph preview shows the existing render error UI. The user can see the raw source and fix it manually.

## Architecture

```
Create view — left column (existing two-column layout)
─────────────────────────────────────────────────────────────────────────
┌─ Describe your pipeline ───────────────────────────────────────────────┐
│  [textarea #nlInput]                                                    │
│  [ ] Simulate    [✓] Auto-approve                                       │
└─────────────────────────────────────────────────────────────────────────┘
┌─ Generated DOT ─────────────────────────────── [Status] ───────────────┐
│  [textarea #dotPreview]                                                 │
│  <input type="file" id="dotFileInput" accept=".dot" hidden>   ← NEW   │
│  [gen-hint] [📂 Upload .dot] [Cancel (hidden)] [▶ Run Pipeline]        │  ← CHANGED
└─────────────────────────────────────────────────────────────────────────┘

Data flow on file select:
  user clicks "📂 Upload .dot"
    → #dotFileInput.click() (button onclick)
    → native file picker opens (filtered to .dot)
  user selects file
    → onDotFileSelected() fires
      1. clearTimeout(genDebounce); clearInterval(genCountdown)   ← cancel pending NL gen
      2. FileReader.readAsText(file, 'UTF-8')
         onload:
           dotPreview.value = result
           nlInput.value = ''                    ← clear NL prompt (clean state)
           uploadedFileName = file.name          ← module-scope var
           runBtn.disabled = !result.trim()
           setGenStatus('ok', 'Loaded: ' + file.name)
           renderRetries = 0
           renderGraph()                         ← re-uses existing function
         onerror:
           setGenStatus('error', 'Could not read file.')

runGenerated() (modified):
  fileName: uploadedFileName || 'generated.dot'   ← use uploaded name if set
  success: uploadedFileName = null                 ← reset after submit

clearCreateForm() (modified):
  dotFileInput.value = ''
  uploadedFileName = null

resetCreatePage() (modified):
  dotFileInput.value = ''
  uploadedFileName = null
```

**Files:**
- `src/main/kotlin/attractor/web/WebMonitorServer.kt` — Modify

**Tasks:**
- [ ] Add `var uploadedFileName = null;` near the top of the script block, alongside other module-scope variables (near `var iterateSourceId = null`).

- [ ] Implement `onDotFileSelected()`:
  ```javascript
  function onDotFileSelected() {
    var input = document.getElementById('dotFileInput');
    if (!input || !input.files || input.files.length === 0) return;
    var file = input.files[0];
    // Cancel any pending NL auto-generate to prevent overwriting uploaded DOT
    clearTimeout(genDebounce);
    clearInterval(genCountdown);
    var reader = new FileReader();
    reader.onload = function(e) {
      var dotSource = e.target.result || '';
      var preview = document.getElementById('dotPreview');
      var nlInput = document.getElementById('nlInput');
      if (preview) {
        preview.value = dotSource;
        if (nlInput) nlInput.value = '';         // clear NL prompt for clean state
        uploadedFileName = file.name;
        document.getElementById('runBtn').disabled = !dotSource.trim();
        setGenStatus('ok', 'Loaded: ' + file.name);
        renderRetries = 0;
        renderGraph();
      }
    };
    reader.onerror = function() {
      setGenStatus('error', 'Could not read file.');
    };
    reader.readAsText(file, 'UTF-8');
  }
  ```

- [ ] Modify `runGenerated()` to use `uploadedFileName` when set:

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Pending NL auto-generate overwrites uploaded DOT | Medium | High | Cancel `genDebounce`/`genCountdown` timers at the top of `onDotFileSelected()` |
| `uploadedFileName` leaks into a subsequent NL-generated pipeline | Medium | Low | Both `clearCreateForm()` and `resetCreatePage()` reset it explicitly |
| File picker accepts non-DOT files | Low | Low | `accept=".dot"` is advisory; browsers enforce it for normal selection. Render errors from invalid DOT are handled gracefully by `renderGraph()` |
| Very large DOT files cause browser lag | Low | Low | DOT files are plain text, rarely >100 KB; `FileReader.readAsText` is async |
| Upload button visually crowds the run-row | Low | Low | Button is small (tertiary style); run-row already has Cancel (hidden by default) and Run Pipeline |

## Security Considerations

- File contents are read client-side by `FileReader` and placed into a textarea. No file data is sent to the server until the user explicitly clicks Run Pipeline — and only then as `dotSource` in the existing `/api/run` JSON body.
- No new server-side attack surface.
- `setGenStatus()` uses `textContent`, not `innerHTML`, so filename cannot inject HTML/script.
- DOT source is already fully user-controlled (users can type anything in the textarea). Accepting a file upload does not change the server-side trust model.
